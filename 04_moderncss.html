Modern CSS

Selector performance

Selector parsing speed

  • keep your selectors performant; some facts:
    • selector types (roughly), from fast to slow:
      • id selector, e.g. #content (fast)
      • class selector, e.g. .button-ok (fast)
      • tag selector, e.g. div (slow)
      • universal selector, e.g. *, [title="test"] (really slow)
    • also slow are adjacent, child and nested selector, e.g. li > a, h1 + p, header p
    • browsers parse selectors from right to left => the right part should be fast
    • every step is checked, so keep selectors as short as possible
Quote from csswizardry.com:
It’s more efficient for a browser to start at the right-most element (the one it knows it wants to style) and work its way back up the DOM tree than it is to start high up the DOM tree and take a journey down that might not even end up at the right-most selector–also known as the key selector.

Overqualification; classes

  • avoid unnecessary checks; don't overqualify selectors 
    div#content { }
p.message { }
span.message { }

    #content { }
.p-message { }
.span-message { }
  • class selectors are faster and more versatile 
    #social_links a { }
input[type=button].ok { }

    .link-social
.button-ok { }
Some explanations:
  • div#content: it performs an unnecessary check to see if it is a div
  • #social_links a: it needs to grab all links in the DOM, and then check the context of each one

Short & inherited

  • use inheritance where-ever possible 
    #mytable td { color: blue; }
    #mytable { color: blue; }
  • keep your selectors short 
    nav #mainmenu li a { }

    #mainmenu a { }
keeping selectors short also has the advantage of keeping selector weight in hand, making future styles cascading easier

Avoid universal selectors

  • avoid ending with a tag or universal selector 
    #mainmenu a { }
input[type=submit]
    .menu-link { }
.button-submit { }
  • is all this necessary?
    • small mostly static sites: probably not
    • sites with lots of dynamic css: probably
    • the next Amazon or Facebook website: definitely

Class selectors

Advantages

  • Id attributes in HTML are fine and sometimes necessary for Javascript. For CSS however, class attributes are preferred:
    • performance difference is almost non-existent
    • ID attributes makes cascading styles more difficult due to increased specificity
    • styles attached to ID attributes cannot be reused
  • let's illustrate this with an example

Example 1 (1)

  • typical HTML/CSS for a horizontal menu: 
    <nav id="mainmenu">
  <ul>
    <li><a href="#">link1</a></li>
    <li><a href="#">link2</a></li>
    <li><a href="#">link3</a></li>
  </ul>
</nav>
    #mainmenu ul {
  margin: 0;
  padding: 0;
  list-style-type: none;
}

#mainmenu li {
  display: inline-block;
  margin-left: 10px;
}
  • can we rewrite this in a more generic, reusable way?

Example 1 (2)

  • rewritten with generic class names: 
    <nav>
  <ul class="l-grid">
    <li class="l-grid-col"><a href="#">link1</a></li>
    <li class="l-grid-col"><a href="#">link2</a></li>
    <li class="l-grid-col"><a href="#">link3</a></li>
  </ul>
</nav>
    .l-grid {
  margin: 0;
  padding: 0;
  list-style-type: none;
}

.l-grid-col {
  display: inline-block;
  margin-left: 10px;
}
    demo
The l- prefix is used here to indicate it's a layout style, but isn't necessary.

Example 1 (3)

  • What did we win / lose:
    • our .l-grid style can be applied to any container with a horizontal layout
    • using class- instead of id-attributes reduces specificity
    • applying styles to your HTML is much easier
    • styles are instantly visible in your HTML
    • you need a few extra bytes for the class attributes
  • all major CSS frameworks are built this way (Bootstrap, Foundation...)

Summing it up

  • to sum things up, pro's & cons of class selectors:
    • fast
    • can be reused
    • reduce specificity, thus making style cascading easier
    • allows writing modular / inherited CSS
    • with proper class naming, styles are instantly visible in your HTML
    • presentational classnames in HTML break separation content–presentation
    • slightly bloated HTML code due to many classnames
  • still, pro's outweight the cons; start thinking in class selectors

Modern CSS

Naming conventions

Naming conventions

  • ok, so we mainly use classes — but how do we name them?
  • classnames are traditionally in dash notation, e.g. .card-title, .btn-group-vertical etc...
  • several naming conventions exist, all more or less following the parent-child-theme structure, the most important being:
    • BEM: Block Element Modifier
    • SMACSS: Scalable and Modular Architecture for CSS
    • OOCSS: Object Oriented CSS
  • whichever system you choose isn't important, as long as it is consistent
Why dashes and not underscores? There isn't a clear answer, but some suggestions:
  • in CSS1, underscores technically needed to be escaped, e.g. p.urgent\_note {color: maroon;}
  • dashes are also used elsewhere in CSS, e.g. :first-child
  • dashes are also used in URL's as Google SEO treats word separated by dashes as separate words; maybe this has also bled in CSS naming conventions?

BEM

  • BEM (Block, Element, Modifier) is the mother of all naming conventions: .block__element--modifier
    • Block: parent; top level element
    • Element: child; is only relevant within the parent
    • Modifier: state or theme

BEM example 1

  • illustration found on this codepen: 
    <a href="#" class="btn">

  <span class="btn__price">$4</span>
  <span class="btn__text">Big button</span>

/* Elements */

.btn__text  { ... }

/* Modifiers */
.btn--big  { ... }
.btn--blue  { ... }
.btn--orange  { ... }

BEM pro's and cons

  • BEM has some nice pro's:
    • quickly see which modifiers and children styles already exist for a block
    • quickly see which element is dependent on another
    • consistent naming => easier communication in a team
    • everything is a class, nothing is nested => fast and flat CSS with low specificity
  • and some con's too:
    • double dashes and underscores might not be everyone's liking
    • you'll need lots of class attributes cluttering your HTML
  • more BEM examples on csswizardry.com

Other naming conventions

  • you don't need to like BEM; an alternative could be: 
    /* element */
.product { ... }

/* child */
.product-title { ... }
.product-btnOrder { ... }

/* modifier, generic */
.product-btnOrder.blue { ... }
.product-btnOrder.orange { ... }

/* modifier, specific */
.product_highlighted { ... }
.product_unavailable { ... }

/* state */
.product.is-selected { ... }
  • at least use some set of rules (with focus on flat CSS and class selectors)
  • don't get paranoïa, they're only guidelines; you can still use other types of selectors (contextual, tag etc...) in specific situations

Other good practices

'Best' practices (1)

  • you'll find lots of best practices on the web, many of them questionable or obsolete
  • csslint for example suggests a fallback color for older browsers: 
    .box {
  background: #000;
  color: blue; /* fallback color for IE8 and lower */
  color: rgba(100, 100, 200, 0.5);
}
  • it's safe to ignore anything below IE9 / IE10
Unless you insist on making your site 100% compatible with IE7-, you don't need hacks. Hacks are generally a sign of poor CSS skills. Don't compensate an error by adding another.

'Best' practices (2)

  • csslint will also warn against zero values with unit because it saves bytes (aww c'mon!) 
    .mybox1 {
  padding: 10px 0px; /* plays nicely with de- and increment in webdev tools */
}
.mybox2 {
    width: 0%; /* same here */
}

    .mybox1 {
  padding: 10px 0; /* you saved 2 bytes, big deal */
}
.mybox2 {
    width: 0; /* you saved 1 byte, wow, really */
}
  • use zeros with or without units, as you like
Unless you insist on making your site 100% compatible with IE7-, you don't need hacks. Hacks are generally a sign of poor CSS skills. Don't compensate an error by adding another.

don't use hacks

  • Unless you insist on making your site 100% compatible with IE7-, you don't need hacks. Hacks are generally a sign of poor CSS skills. Don't compensate an error or bug by adding another. 
    .mybox {
  border: 1px solid black;
  padding: 5px;
  width: 100px;
  _width: 200px; /* IE7- only */
}

Use shorthand notation

  • shorthand notation reduces file size and improves overall readability 
    #mybox {
  margin-top: 20px;
  margin-right: 10px;
  margin-bottom: 20px;
  margin-left: 10px;
}
    #mybox {
  margin: 20px 10px;
}

avoid !important

  • don't use !important to artificially increase specificity: 
    .mybox {
    color: red !important;
}
  • two acceptable cases:
    • to overrule inline CSS, e.g. generated by Javascript (no alternative there)
    • to tweak an existing theme, e.g. Wordpress
Using !important is usually an indication that the specificity of the entire CSS has gotten a bit out of control and needs to be refactored.

more good practices

  • other advices:
    • do not use a CSS framework like Bootstrap or Foundation to hide poor CSS skills; be a Vanilla CSS expert at all times
    • favor class selectors over id, tag or other selectors
    • minify and concatenate your CSS files
    • find a way to organize your stylesheets, e.g. top-down
      • generic styles (clearfix, html fontsize...)
      • general styles (body, a, p...)
      • header
        • layout
        • logo
        • menu
        • ...
      • etc...
    • comment each block
    • consider separating skin (fonts, colors...) from structure in separate files
    • above all: be consistent

SASS

About SASS

  • Sass (Syntactically Awesome Style Sheets) enhances Vanilla CSS:
    • calculations (actually currently better in Vanilla CSS)
    • variables (bad native browser support and performance issues in Vanilla CSS)
    • imports (@import in Vanilla CSS has performance issues due to HTTP requests)
    • mixins (not considered in Vanilla CSS, probably performance issues)
    • selector nesting (doesn't exist in Vanilla CSS)
    • if-else & for (doesn't exist in Vanilla CSS)
    • ...
  • in Sass, values are calculated at compile time, which has pro's (e.g. performance) and con's (not dynamic: think Javascript, values like 100% - 20px etc...)

SASS or SCSS syntax

  • older SASS syntax: 
    nav
  li
    float: left

    + li::before
      content: " | "

    a
      display: block
      line-height: 1.3
      padding: 0 0.5rem
  • newer SCSS syntax: 
    nav {
  li {
    float: left;

    + li::before {
      content: " | ";
    }

    a {
      display: block;
      line-height: 1.3;
      padding: 0 0.5rem;
    }
  }
}
  • SCSS syntax is an extension of CSS (recommended)
  • tip: try your SCSS out on the online scss compiler

nesting

  • nesting in SCSS creates contextual selectors: 
    nav {
  li {
    float: left;

    + li::before {
      content: " | ";
    }

    a {
      display: block;
      line-height: 1.3;
      padding: 0 0.5rem;
    }
  }
}
  • rendered: 
    nav li {
  float: left;
}
nav li + li::before {
  content: " | ";
}
nav li a {
  display: block;
  line-height: 1.3;
  padding: 0 0.5rem;
}

  • control structures like @if, @for... in SCSS:

partials & imports (2)

  • use partials to organize your CSS in separate files: 
    partials/
  |-- mixins/
    |-- _animations.scss
    |-- _clearfix.scss
    |-- _gradients.scss
    |-- ...
  |-- _base.scss
  |-- _forms.scss
  |-- _grids.scss
  |-- _icons.scss
  |-- _menus.scss
  |-- _messages.scss
  |-- _panes.scss
  |-- _reset.scss
  |-- _typography.scss
  `-- ...
  • your main file is kept extremely clean: 
    // modules and variables
@import "partials/base";

// partials
@import "partials/reset";
@import "partials/typography";
@import "partials/icons";
// ...
You could theoretically also use CSS's native @import but this generates an extra HTTP request. As a modern site will easily need dozens of imports, this is detrimental for your site performance.

mixins

  • a mixin is a reusable function generating a block of SCSS: 
    @mixin my_links($link, $hover, $size: 1.4rem) {
  a {
    color: $link;
    font-size: $size;
    &:hover {
      color: $hover;
    }
  }
}

nav {
  @include my_links(#333, #999);
}

footer {
  @include my_links(#eee, #ccc, 1.2rem);
}

    nav a {
  color: #333;
  font-size: 1.4rem;

  color: #999;
}

  color: #eee;
  font-size: 1.2rem;

  color: #ccc;

calculations

  • SCSS offers basic calculations: 
    .container {
  width: 600px / 960px * 100%; /* 62.5% */
}
  • note: like all preprocessors, it can only calculate static values: 
    .container {
  width: 100% - 2rem; /* will throw an error */
}
  • SCSS has more: custom functions, interpolations etc..., see documentation

Pitfalls (1)

  • you'll often see SCSS code like this, compiling nicely to BEM: 
    .product {
  &--highlight {
    background-color: #ffc;
  }
  &__btnOrder {
    border-color: #9f9;
    &--minimal {
      background-color: transparent;
    }
  }
}
    .product--highlight {
  background-color: #ffc;
}
.product__btnOrder {
  border-color: #9f9;
}
.product__btnOrder--minimal {
  background-color: transparent;
}
  • your code is not searcheable anymore for e.g. .product__btnOrder
  • don't use & to generate new class names

Pitfalls (2)

  • beware of deep nesting: 
    .nav {
  float: right;
  ul {
    overflow: hidden;
    li {
      display: inline-block;
      .nav-link {
        text-decoration: none;
        &:hover {
          text-decoration: underline;
        }
      }
    }
  }
}
    ...
.nav ul li .nav-link:hover {
  text-decoration: underline;
}
  • the compiled CSS selectors are long and inefficient (slow)
  • keep your SCSS flat and simple, just like CSS
  • how would you correct this?

Answer:

.nav {
  float: right;
  ul {
    overflow: hidden;
  }
  li {
    display: inline-block;
  }
  .nav-link {
    text-decoration: none;
    &:hover {
      text-decoration: underline;
    }
  }
}

CSS frameworks

Frontend Frameworks

  • frontend frameworks offer methodology and/or rich components:
    • CSS frameworks focus on design; Javascript is mostly used for component behaviour (e.g. accordeon) -> Bootstrap, Foundation, MDL
    • Javascript frameworks focus on coding methodology; design is left to the programmer -> Angular, React
    • full frontend frameworks offer design and methodology, e.g. Polymer
  • when choosing a framework, watch out for
    • popularity (tutorials, user base, third party code), is it future-proof?
    • maturity; how likely are major changes?
    • specificity (learning curve)
    • good documentation

Still, chosing a future-proof framework is a hit and miss. Backbone was the golden standard a few years ago, but today it's obsolete. The immensely popular Angular completely overthrew its syntax in version 2, which is not compatible with version 1.

As for specificity: that's why SCSS is better than SASS. It extends the CSS syntax instead of creating its own like SASS, resulting in a smoother learning curve.

Bootstrap (1)

  • Bootstrap is no doubt the most popular CSS framework today.
  • It allows you to throw together a standard interface in just minutes:
    1. install bootstrap with Bower or via Bootstrap CDN
    2. start with this basic template or from these examples
    3. create your responsive grid (based on 12 columns)
    4. toss in the components you need
    5. choose a theme, create one, customize or extend the SCSS
  • be sure to use Bootstrap 4 (in alpha now), not Bootstrap 3:
    • uses SCSS instead of LESS
    • uses rem instead of px
    • improved grid system
    • ...

Bootstrap (2)

  • Bootstrap's most important feature is without any doubt its responsive grid system
  • Bootstrap 4 uses 5 viewport widths:
    • -xs-: <544px
    • -sm-: ≥544px
    • -md-: ≥768px
    • -lg-: ≥992px
    • -xl-: ≥1200px

Material Design (1)

  • Material Design is not a CSS framework, but a design philosophy
  • MD is meant to be the de facto standard for Google products (e.g. maps) and Android apps
  • MD is arguably the most comprehensive design guideline today:
    • guidelines about basically everything: colors, animations, layout, components...
    • lightning/shadows, surface, movements etc. are based on based on paper properties (not always possible, e.g. expanding areas)

Material Design (2)

  • to illustrate, let's zoom in on one of the many aspects of MD: material properties
  • rule 1: material has a fixed thickness

Material Design (3)

  • rule 2: no bending or folding

Material Design (4)

  • rule 3: material cannot pass through another material

Material Design (5)

  • rule 4: material can split and heal

Material Design (6)

  • rule 5: elevation is decided by its function and state

Material Design (7)

  • Many frameworks are based on Material Design:
    • Materialize: CSS framework; feels like Bootstrap
    • Material Design Lite: CSS framework; BEM-based
    • Google Polymer: frontend framework
    • Angular Material: frontend framework; Angular's implementation of MD; version for Angular 2 not ready yet
  • great comparison between Material Design Lite and Bootstrap (3) on tutorialzine.com with many code examples

Final thoughts

  • You'll have noticed some striking limitations and differences in Bootstrap's and MDL's grid system:
    • both have fixed breakpoint values
    • Bootstrap has 5 breakpoints, MDL 3
    • MDL has a 8px gutter between cells, Bootstrap doesn't
    • Bootstrap uses the concept of rows, MDL doesn't
  • A CSS framework has a certain style. Don't hack or fight it.
apart from basic theming, don't twist or bend a framework; embrace it's style, choose another or use Vanilla CSS; for custom tailored designs, a framework is probably not the answer
